SFQA PLAYWRIGHT ARCHITECTURE

1. HOW DATA FLOWS (ONE-WAY ONLY)
[Dashboard UI (pages/tests.js)] ---> [Database (data/tests.json)] ---> [Code Generator (lib/dataStore.js)] ---> [Script (tests/*.spec.js)]
* Rule: Dashboard creates script. Script does NOT create Dashboard. 

2. MAIN FILES
- pages/tests.js: The UI you see.
- data/tests.json: The database saving your test steps.
- lib/dataStore.js: The machine that translates JSON database into Playwright code.
- pages/api/trigger-run.js: The runner that tells Playwright to execute the code.
- public/chrome_extension: The spy that watches your clicks and sends them to the UI.

3. HOW RUN WORKS
1. You click "Run".
2. Dashboard tells trigger-run.js to start.
3. It spawns Playwright in background.
4. Logs stream back to UI.

4. FOLDER RESPONSIBILITIES
- /pages: Next.js frontend pages (UI) and API routes (backend endpoints). Responsible for functions and javascript behaviors.
- /public: Static assets (images, icons) and the Chrome extension source code.
- /components: Reusable React UI components (like Modals).
- /data: Stores JSON database files (e.g., tests.json, runs.json).
- /lib: Core backend logic, utilities, and the Playwright code generator (dataStore.js).
- /styles: Global CSS stylesheets.
- /tests: Auto-generated Playwright test scripts (.spec.js files).
- /test-results: Output folder for Playwright test results, traces, and reports.
- /DOCS: Project documentation, standard operating procedures, and architecture notes.
